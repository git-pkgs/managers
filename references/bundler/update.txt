PPAATTCCHH LLEEVVEELL OOPPTTIIOONNSS
       Version 1.14 introduced 4 patch-level options that will influence how
       gem versions are resolved. One of the following options can be used:
       ----ppaattcchh, ----mmiinnoorr or ----mmaajjoorr. ----ssttrriicctt can be added to further influence
       resolution.

       ----ppaattcchh
              Prefer updating only to next patch version.

       ----mmiinnoorr
              Prefer updating only to next minor version.

       ----mmaajjoorr
              Prefer updating to next major version (default).

       ----ssttrriicctt
              Do not allow any gem to be updated past latest ----ppaattcchh | ----mmiinnoorr
              | ----mmaajjoorr.

       When Bundler is resolving what versions to use to satisfy declared
       requirements in the Gemfile or in parent gems, it looks up all
       available versions, filters out any versions that don't satisfy the
       requirement, and then, by default, sorts them from newest to oldest,
       considering them in that order.

       Providing one of the patch level options (e.g. ----ppaattcchh) changes the
       sort order of the satisfying versions, causing Bundler to consider the
       latest ----ppaattcchh or ----mmiinnoorr version available before other versions. Note
       that versions outside the stated patch level could still be resolved to
       if necessary to find a suitable dependency graph.

       For example, if gem 'foo' is locked at 1.0.2, with no gem requirement
       defined in the Gemfile, and versions 1.0.3, 1.0.4, 1.1.0, 1.1.1, 2.0.0
       all exist, the default order of preference by default (----mmaajjoorr) will be
       "2.0.0, 1.1.1, 1.1.0, 1.0.4, 1.0.3, 1.0.2".

       If the ----ppaattcchh option is used, the order of preference will change to
       "1.0.4, 1.0.3, 1.0.2, 1.1.1, 1.1.0, 2.0.0".

       If the ----mmiinnoorr option is used, the order of preference will change to
       "1.1.1, 1.1.0, 1.0.4, 1.0.3, 1.0.2, 2.0.0".

       Combining the ----ssttrriicctt option with any of the patch level options will
       remove any versions beyond the scope of the patch level option, to
       ensure that no gem is updated that far.

       To continue the previous example, if both ----ppaattcchh and ----ssttrriicctt options
       are used, the available versions for resolution would be "1.0.4, 1.0.3,
       1.0.2". If ----mmiinnoorr and ----ssttrriicctt are used, it would be "1.1.1, 1.1.0,
       1.0.4, 1.0.3, 1.0.2".

       Gem requirements as defined in the Gemfile will still be the first
       determining factor for what versions are available. If the gem
       requirement for ffoooo in the Gemfile is '~> 1.0', that will accomplish
       the same thing as providing the ----mmiinnoorr and ----ssttrriicctt options.

PPAATTCCHH LLEEVVEELL EEXXAAMMPPLLEESS
       Given the following gem specifications:


           foo 1.4.3, requires: ~> bar 2.0
           foo 1.4.4, requires: ~> bar 2.0
           foo 1.4.5, requires: ~> bar 2.1
           foo 1.5.0, requires: ~> bar 2.1
           foo 1.5.1, requires: ~> bar 3.0
           bar with versions 2.0.3, 2.0.4, 2.1.0, 2.1.1, 3.0.0




       Gemfile:


           gem 'foo'




       Gemfile.lock:


           foo (1.4.3)
             bar (~> 2.0)
           bar (2.0.3)




       Cases:


           #  Command Line                     Result
           ------------------------------------------------------------
           1  bundle update --patch            'foo 1.4.5', 'bar 2.1.1'
           2  bundle update --patch foo        'foo 1.4.5', 'bar 2.1.1'
           3  bundle update --minor            'foo 1.5.1', 'bar 3.0.0'
           4  bundle update --minor --strict   'foo 1.5.0', 'bar 2.1.1'
           5  bundle update --patch --strict   'foo 1.4.4', 'bar 2.0.4'




       In case 1, bar is upgraded to 2.1.1, a minor version increase, because
       the dependency from foo 1.4.5 required it.

       In case 2, only foo is requested to be unlocked, but bar is also
       allowed to move because it's not a declared dependency in the Gemfile.

       In case 3, bar goes up a whole major release, because a minor increase
       is preferred now for foo, and when it goes to 1.5.1, it requires 3.0.0
       of bar.

       In case 4, foo is preferred up to a minor version, but 1.5.1 won't work
       because the --strict flag removes bar 3.0.0 from consideration since
       it's a major increment.

       In case 5, both foo and bar have any minor or major increments removed
       from consideration because of the --strict flag, so the most they can
       move is up to 1.4.4 and 2.0.4.

       In general, when working with an application managed with bundler, you
       should use the following workflow:

       •   After you create your Gemfile(5) for the first time, run

           $ bundle install

       •   Check the resulting GGeemmffiillee..lloocckk into version control

           $ git add Gemfile.lock

       •   When checking out this repository on another development machine,
           run

           $ bundle install

       •   When checking out this repository on a deployment machine, run

           $ bundle install --deployment

       •   After changing the Gemfile(5) to reflect a new or update
           dependency, run

           $ bundle install

       •   Make sure to check the updated GGeemmffiillee..lloocckk into version control

           $ git add Gemfile.lock

       •   If bundle install(1) _b_u_n_d_l_e_-_i_n_s_t_a_l_l_._1_._h_t_m_l reports a conflict,
           manually update the specific gems that you changed in the
           Gemfile(5)

           $ bundle update rails thin

       •   If you want to update all the gems to the latest possible versions
           that still match the gems listed in the Gemfile(5), run

           $ bundle update --all
